- **인스타그램**: @hyuk.xyz

### 작업 성향
- 추상적인 주제를 **고유한 논리, 규칙, 매뉴얼**로 분류·분석하여 작업물로 전환한다
- **"최신의 기술로 최고의 아날로그함을 추구"** — 기술과 아날로그 감성의 교차점을 탐구한다

### 리서치 방향
- 디자인 작업 자체의 리서치가 아니라, **사회적 현상**을 자신만의 방식으로 리서치한다
- 리서치 결과는 궁극적으로 디자인 작업의 개념적 토대가 된다
- Claude는 사회·문화 현상의 조사 파트너이며, 디자인 실무 도구가 아니다

## 이 리포지토리의 목적
이 리포지토리는 그래픽 디자이너의 개인 리서치 랩입니다.
사회적 현상과 추상적 주제를 체계적으로 조사·분류하고, 이를 작업의 개념적 기반으로 활용하기 위한 공간입니다.

## 작업 원칙

### 리서치 워크플로우
1. 아이디어나 질문이 떠오르면 `inbox/`에 빠르게 캡처한다
2. 본격적인 리서치는 `projects/YYYY-MM-DD_주제명/` 폴더를 만들어 진행한다
3. 각 프로젝트 폴더에는 `_index.md`(MOC/요약)와 `findings.md`(상세 내용)를 포함한다
4. 지속적으로 관심 있는 분야는 `areas/`에서 관리한다
5. 참고 자료, 문헌 노트는 `resources/`에 정리한다
6. 완료된 프로젝트는 `archive/`로 이동한다
7. 주기적으로 `inbox/`를 정리하여 적절한 위치로 이동한다

### YAML 프론트매터 규칙
모든 마크다운 문서에 아래 형식의 프론트매터를 포함한다:
```yaml
---
title: "제목"
date: YYYY-MM-DD
updated: YYYY-MM-DD
status: inbox | in-progress | review | complete | archived
type: research | literature-note | concept | reference | log
tags: [태그1, 태그2]
related: ["파일경로1.md", "파일경로2.md"]
confidence: high | medium | low | speculative
---
```

### 태깅 규칙
- 태그는 반드시 YAML 프론트매터의 `tags` 필드에 배열로 작성한다 (본문 내 `#태그` 사용 금지)
- kebab-case 사용: `machine-learning`, `large-language-model`
- `_tags.md` 파일에 공식 태그 목록을 관리하여 태그 중복/분산을 방지한다

### 링크 규칙
- 표준 마크다운 링크를 사용한다: `[제목](상대경로.md)`
- 항상 상대 경로를 사용한다 (절대 경로 금지)
- 각 노트 하단에 "관련 문서" 섹션을 두어 2~5개의 관련 노트를 링크한다

### 품질 기준
- 출처가 있는 정보는 반드시 출처를 명시한다
- 주장과 사실을 구분하여 기술한다
- 불확실한 정보는 `confidence: low` 또는 `confidence: speculative`로 표시한다
- 한국어로 작성하되, 기술 용어는 영어 병기를 허용한다

### 결과물 형식
- 마크다운(.md)을 기본 형식으로 사용한다
- 코드 예제가 포함될 경우 적절한 언어 태그를 사용한다
- 다이어그램이 필요한 경우 Mermaid 문법을 사용한다
- 이미지/PDF 등 바이너리 파일은 `assets/` 디렉토리에 저장한다

### Maps of Content (MOC)
- 각 주요 폴더에 `_index.md` 파일을 두어 해당 폴더의 네비게이션 허브로 사용한다
- MOC는 단순 목록이 아니라, 맥락과 설명을 포함한 큐레이션된 링크 페이지이다
- 특정 주제에 노트가 5개 이상 쌓이면 해당 주제의 MOC를 생성한다

### Claude 협업 규칙
- Claude는 "생각 모드"(질문하고 토론)와 "작성 모드"(결과물 생성)를 구분한다
- 리서치 시작 시 먼저 핵심 질문을 정리하고 접근 방법을 논의한 후 조사에 들어간다
- Claude는 새 노트 작성 시 관련 기존 노트와의 연결을 제안한다
- 최종 링크/분류 결정은 사용자가 한다

### 비평 프로토콜 (Red Team)
리서치가 방향을 잃거나 불필요하게 비대해지는 것을 방지하기 위한 구조적 장치.

**`_critique.md` 파일**: 각 프로젝트 폴더에 `_critique.md`를 둔다. 이 파일은 리서치 내용을 확장하는 것이 아니라 **검증하고 제한하는** 역할을 한다.

**작성 시점**: 아래 상황에서 Claude가 자동으로 `_critique.md`를 업데이트한다:
- 프로젝트에 새로운 섹션이 추가될 때
- 다른 프로젝트와 연결이 생길 때
- 사용자가 방향 전환을 요청할 때

**필수 항목** (매 업데이트마다 갱신):
```markdown
## 핵심 주장
이 리서치의 핵심 주장을 한 문장으로. 못 쓰면 아직 모호한 것.

## 범위 경계
이 리서치가 다루는 것 / 다루지 않는 것을 명시적으로 구분.

## 가장 약한 고리
현재 리서치에서 근거가 가장 부족하거나 논리적으로 가장 취약한 부분.

## 반론
이 방향이 틀렸다면 가장 가능성 높은 이유.

## 복잡도 경고
현재 리서치가 감당할 수 없을 만큼 커지고 있는가? 쳐내야 할 것은?

## 실행 가능성
현재 환경(아이폰, 카메라 불가)에서 실제로 시도할 수 있는 것이 있는가?
```

**Claude의 비평 태도**:
- 사용자의 방향에 동의하며 자료를 추가하는 것이 기본이 아니다
- 매 리서치 확장 시 "이것이 정말 필요한가?"를 먼저 묻는다
- 매력적이지만 근거 없는 연결(예: "원래 하나였다" 식의 낭만적 주장)에 명시적으로 제동을 건다
- 사용자가 명시적으로 "확장해달라"고 요청한 경우에도, 확장 후 비평을 반드시 갱신한다

## 디렉토리 구조

```
researchlab/
├── CLAUDE.md              # 이 파일 — Claude 협업 규칙
├── README.md              # 리서치 랩 소개
├── _index.md              # 최상위 MOC (전체 네비게이션)
├── _tags.md               # 공식 태그 목록
├── inbox/                 # 빠른 캡처 (정리 전 메모, 아이디어)
├── projects/              # 진행 중인 리서치 프로젝트
│   └── YYYY-MM-DD_주제명/
│       ├── _index.md      # 프로젝트 MOC
│       ├── findings.md    # 상세 조사 내용
│       ├── sources.md     # 출처 정리
│       └── _critique.md   # 비평 (Red Team)
├── areas/                 # 지속적 관심 분야 (마감 없음)
├── resources/             # 참고 자료, 문헌 노트
├── archive/               # 완료/비활성 프로젝트
├── templates/             # 노트 템플릿
└── assets/                # 이미지, PDF 등 첨부 파일
    └── images/
```

## 서브에이전트 사용 규칙

웹 리서치를 서브에이전트에 위임할 때 반드시 아래 규칙을 따른다:

### 실패 즉시 포기 (Fail Fast)
- **WebFetch/WebSearch에서 403, 429, 5xx 에러가 발생하면 해당 URL/쿼리를 즉시 포기**한다. 재시도하지 않는다.
- 하나의 소스가 차단되면, 차단된 소스를 우회하려 하지 말고 다른 소스로 이동하거나 현재까지 수집한 결과를 반환한다.
- 3회 이상 연속 에러 발생 시, 웹 리서치 자체를 중단하고 수집된 결과만으로 응답한다.

### 작업 단위 분리
- 서브에이전트 1개당 리서치 주제는 **최대 2개**까지만 할당한다.
- 3개 이상의 주제를 조사해야 할 경우, 여러 에이전트를 병렬로 실행한다.
- 각 에이전트의 프롬프트에는 위 "실패 즉시 포기" 규칙을 명시적으로 포함한다.

### 프롬프트 필수 포함 문구
서브에이전트에 웹 리서치를 위임할 때, 프롬프트 끝에 반드시 아래 문구를 포함한다:
```
[에이전트 규칙] WebFetch/WebSearch에서 403·429·5xx 에러 발생 시 해당 소스를 즉시 포기하고 다른 소스로 이동하라. 3회 연속 에러 시 웹 리서치를 중단하고 현재까지 수집한 결과만 반환하라. 결과가 부족해도 타임아웃보다 낫다.
```

## 커밋 메시지 규칙
- `research: 주제명 - 내용` — 리서치 관련
- `notes: 내용` — 메모/노트 관련
- `docs: 내용` — MOC, 인덱스 업데이트
- `chore: 내용` — 설정/구조 변경
